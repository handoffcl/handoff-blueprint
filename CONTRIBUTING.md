## Español

¡Gracias por querer contribuir a Handoff Blueprint!

### Cómo contribuir

- **Reportar un problema** — Abre un Issue describiendo el problema con detalle
- **Proponer una mejora** — Abre un Issue con el tag `enhancement` antes de codear
- **Enviar un PR** — Fork → rama descriptiva → PR con contexto claro

### Qué puedes mejorar

- Templates de `CONTEXT.md`, `WORKING-AGREEMENT.md` o `CLAUDE.md`
- Comandos `/bootstrap-app` o nuevos slash commands
- Ejemplos de proyectos reales usando el blueprint
- Traducciones y documentación

### Reglas básicas

1. Un PR = un cambio concreto. Sin refactors mezclados.
2. Describe el **por qué**, no solo el qué.
3. Si es un cambio grande, abre un Issue primero.
4. Respeta el tono y estructura existente del blueprint.
5. **Cambios en `commands/` o `blueprint/` afectan la extensión VS Code directamente** — coordina con los maintainers antes de modificarlos.

## English

Thanks for wanting to contribute to Handoff Blueprint!

### How to contribute

- **Report a bug** — Open an Issue describing the problem in detail
- **Propose an improvement** — Open an Issue tagged `enhancement` before coding
- **Submit a PR** — Fork → descriptive branch → PR with clear context

### What you can improve

- Templates for `CONTEXT.md`, `WORKING-AGREEMENT.md` or `CLAUDE.md`
- The `/bootstrap-app` command or new slash commands
- Real project examples using the blueprint
- Translations and documentation

### Basic rules

1. One PR = one concrete change. No mixed refactors.
2. Describe the **why**, not just the what.
3. For large changes, open an Issue first.
4. Respect the existing tone and structure of the blueprint.
5. **Changes to `commands/` or `blueprint/` directly affect the VS Code extension** — coordinate with maintainers before modifying them.
